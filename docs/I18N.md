# Internationalization (i18n)

Trenerka frontend uses [i18next](https://www.i18next.com/) + [react-i18next](https://react.i18next.com/) with browser language detection.

## Supported languages

| Code | Language | Notes |
|------|----------|-------|
| `ru` | Russian | Default fallback |
| `en` | English | Source for many secondary locales |
| `de` | German | |
| `pt` | Portuguese | |
| `ja` | Japanese | |
| `it` | Italian | |
| `es` | Spanish | |
| `fr` | French | |
| `ar` | Arabic | RTL ŌĆö `dir="rtl"` on `<html>` when selected |
| `zh-CN` | Chinese (Simplified) | |

User choice is stored in `localStorage` under key **`trenerka_lang`**.

## Folder structure

```
frontend/src/i18n/index.ts          # i18n init, supported languages, RTL helper
frontend/src/lib/i18n-format.ts     # Intl date/currency/number by active language
frontend/src/lib/exercise-i18n.ts   # Muscle/equipment/exercise display helpers
frontend/src/locales/{lang}/        # Per-locale JSON namespaces
  common.json
  landing.json
  auth.json

  client.json
  admin.json
frontend/scripts/generate-locales.mjs

```

## Adding a new string

1. Add the key to **`frontend/scripts/locale-extensions.mjs`** under `ru` and `en` (or edit `generate-locales.mjs` for landing-specific blocks).
2. Run `node frontend/scripts/generate-locales.mjs` to sync all 10 languages.
3. In a component:

```tsx
import { useTranslation } from 'react-i18next'

function MyComponent() {
  const { t } = useTranslation('trainer')
  return <button>{t('clients.actions.message')}</button>
}
```

4. Interpolation: `t('clients.inDatabase', { count: 12 })`.
5. Cross-namespace: `t('common:actions.save')`.

## Language switcher

`LanguageSwitcher` is mounted in:

- Site header (landing / public pages)
- Trainer layout header
- Trainer settings page
- Client shell header
- Admin layout header (mobile + desktop)

## RTL (Arabic)

`applyDocumentDirection()` in `src/i18n/index.ts` sets `document.documentElement.dir` to `rtl` for `ar`, `ltr` otherwise.

## Dates and numbers

Use helpers from `@/lib/i18n-format` (re-exported from `@/lib/utils`):

- `formatDate`, `formatDateTime`, `formatRub`, `formatNumber`, `intlLocale`

They follow the active i18n language, not a hardcoded `ru-RU`.

## Exercise content

Mock/API exercises store Russian canonical values. UI translates via:

- `catalog.muscles.*`, `catalog.equipment.*` in `trainer.json`
- `catalog.exercises.{id}.*` for showcase exercises (e.g. `ex-rdl`)
- `displayExercise()`, `translateMuscle()` from `@/lib/exercise-i18n`

## Regenerating locale files

```bash
cd frontend && node scripts/generate-locales.mjs
```

## Build

```bash
cd frontend && npm run build
```

In development, `debug` and `saveMissing` are enabled on i18next ŌĆö missing keys log to the console. Production build should show no missing-key warnings.

### Intentionally not translated

| Item | Reason |
|------|--------|
| Brand **Trenerka** | Product name |
| `Logo` alt text | Brand |
| Demo user names (e.g. ąÉąĮąĮą░ ąĪą╝ąĖčĆąĮąŠą▓ą░ in files mock) | Proper names |
| Email addresses, URLs, demo credentials | Technical / data |
| `SUPPORTED_LANGUAGES[].native` labels | Language names in native script |
| Mock exercise **names** in API seed (display translated when `catalog.exercises.{id}` exists; generic catalog names stay as stored until full exercise translation table) |
| Chart month abbreviations from mock API (`ąöąĄą║`, `ą»ąĮą▓`) | Data layer; translate when chart labels are wired through i18n formatters |
